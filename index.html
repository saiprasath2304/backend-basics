Backend, generally

The backend is the part of an application that runs on a server, not in the user's browser. It's invisible to the user — there's no UI to click on — but it does the real work: storing data, applying rules, and deciding what each request is allowed to do.

A useful way to think about it: the frontend is the dining room — what the customer sees and interacts with. The backend is the kitchen — where the food is actually prepared, ingredients are stored, and orders get matched to recipes. The customer never walks into the kitchen, but nothing reaches the table without it.

A backend is usually built from a few recurring pieces:

server

A long-running program that listens for incoming requests (usually over HTTP) and responds to them.

router

Decides which piece of code should handle a given request, based on the URL and the HTTP method.

controller / handler function

The actual logic for one specific action — e.g. "fetch all users," or "create a new order."

database

Where data is stored permanently, even after the server restarts.

A couple of distinctions that confuse people early on are worth untangling:

Server vs. server-side code. "Server" can mean a physical (or virtual) machine sitting in a data center, or the program running on it that handles requests. In day-to-day conversation, people usually mean the program — e.g. "restart the server" means "restart the process," not "unplug the computer."

Stateless requests. Most backends are designed so each request is handled independently — the server doesn't "remember" you between requests just because you talked to it a moment ago. If it needs to recognize you, that information (like a login token) has to be sent again with every request. This is part of why REST APIs scale well: any server in a pool can answer any request, since none of them are holding onto a private memory of you.

Synchronous vs. asynchronous work. Some operations finish almost instantly (reading one row from a database). Others take real time (resizing a video, calling an external payment provider). Backends are written to avoid blocking — while one slow request is waiting on something, the server should still be free to handle other requests, not sit frozen.

Architecture patterns

Once a backend grows past a handful of routes, it helps to organize the code by responsibility rather than dumping everything in one file. The most common pattern, and the one this tutorial's flow is loosely based on, is MVC:

Model

Represents one kind of data and how to read/write it — e.g. a User model wraps the users table.

View

What gets shown to the user. In an API-only backend there's often no "view" in the traditional sense — the JSON response plays that role, and the frontend renders the actual view.

Controller

The handler function — receives the request, talks to the model, decides what response to send. This is the "Function" box in the tutorial.

Another piece worth knowing about is middleware — small functions that run before a request reaches its controller, often chained together. Common uses: checking whether a user is logged in, logging every incoming request, or rejecting a request if its body is malformed.

router.get('/users', requireLogin, UserController.getUsers); // requireLogin runs first — if it fails, getUsers never runs

Lastly, most real projects keep configuration — database passwords, API keys, which environment they're running in — out of the code itself, using environment variables. This is why you'll often see a call like process.env.DATABASE_URL instead of a password typed directly into a file. It means the same code can run against a test database locally and a production database on a real server, with nothing in the code changing.

REST APIs

An API (Application Programming Interface) is just an agreed-upon way for two programs to talk to each other. A REST API is a common style of API built on top of HTTP, the same protocol your browser already uses to load web pages.

REST organizes everything around two ideas: a resource (a "thing," like a user, an order, or a comment) and an HTTP method (what you want to do to that thing).

GET /users // list all users GET /users/12 // get one user, id 12 POST /users // create a new user PUT /users/12 // replace user 12 PATCH /users/12 // update part of user 12 DELETE /users/12 // remove user 12

Every response carries a status code — a short number that tells you broadly what happened before you even look at the body:

codemeaningtypical cause
200OKrequest succeeded
201Createda POST successfully made something new
400Bad Requestthe request was malformed, e.g. missing a field
401Unauthorizedyou're not logged in
403Forbiddenyou're logged in, but not allowed to do this
404Not Foundthat resource doesn't exist
500Server Errorsomething broke on the backend, not your fault

The response body is almost always JSON — a lightweight, text-based format for structured data that both JavaScript and most other languages can read easily. Requests also carry headers: small bits of metadata sent alongside the body, like Content-Type: application/json (this is JSON) or Authorization: Bearer ... (here's proof of who I am).

A few more ideas that come up constantly once you're working with real APIs:

idempotency

Calling the same request twice should leave things in the same state as calling it once. DELETE /users/12 twice should not cause an error the second time — the user is just already gone. GET and PUT are expected to be idempotent; POST usually isn't, since calling it twice often creates two records.

pagination

"Give me every order" could mean millions of rows. Real APIs usually return a page at a time, e.g. GET /orders?page=2&limit=50, so neither the database nor the network gets overwhelmed.

versioning

APIs change over time, but old clients (an app that hasn't been updated) may still depend on the old shape. Prefixing routes with a version, like /v1/users, lets you change /v2/users without breaking everyone still on v1.

authentication vs. authorization

Authentication answers "who are you?" (logging in). Authorization answers "are you allowed to do this?" (an admin can delete a user; a regular user can't). A request can be authenticated and still get a 403 if it's not authorized.

Databases & SQL

A database stores data in an organized, durable way, and lets you ask questions of it — "give me every user," "find the order placed yesterday," "count how many comments this post has." A common kind, a relational database, stores data in tables: rows and columns, much like a spreadsheet, where each row is one record and each column is one field.

SQL (Structured Query Language) is how you talk to a relational database.

SELECT id, name, email FROM users WHERE role = 'admin';

In most real backends, you don't write SQL by hand for every action. Instead you use an ORM (Object-Relational Mapper) — a library that lets you write plain code like User.findAll(), which it then translates into SQL for you. It's a convenience layer, not a replacement for understanding what's happening underneath — which is exactly what the tutorial visualizes.

A few ideas that explain a lot once they click:

relationships & foreign keys

Tables connect to each other. An orders table might have a user_id column pointing back to a row in users — that's a foreign key. It's how "one user has many orders" gets represented without copying the user's data into every order row.

JOIN

A query that pulls matching rows from two related tables at once — e.g. "every order, along with the name of the user who placed it" — by following that foreign key.

index

A separate, ordered structure the database keeps on a column so it doesn't have to scan every row to find matches — similar to how a book's index lets you skip straight to a page instead of reading cover to cover. Indexes make reads faster but make writes slightly slower, since the index has to be updated too.

transaction

A group of operations that must all succeed, or all be undone together. If a transfer needs to subtract money from one account and add it to another, a transaction guarantees you never end up with the first half done and the second half failed.

migration

A version-controlled, repeatable script that changes the database's structure — e.g. "add a role column to users" — so every environment (your laptop, a teammate's laptop, production) ends up with the exact same schema.

Not every database is relational. NoSQL databases (like MongoDB or DynamoDB) skip the strict rows-and-columns structure in favor of more flexible documents, often trading some consistency guarantees for speed and flexibility at large scale. Most teams default to a relational database unless they have a specific reason not to — the structure tends to catch mistakes early rather than later.

Security basics

None of this needs to be intimidating, but a few habits separate "it works" from "it's safe to actually ship":

never store passwords as plain text

Passwords are run through a one-way hashing function (like bcrypt) before being saved. The database never holds the actual password — only something derived from it that can verify a login attempt but can't be reversed back into the original.

SQL injection

If user input gets pasted directly into a SQL string, a malicious input like ' OR 1=1 can change what the query actually does. ORMs and parameterized queries prevent this by treating input strictly as data, never as part of the SQL itself.

HTTPS

Encrypts traffic between the client and server, so anyone intercepting it in transit (e.g. on shared wifi) sees scrambled data instead of readable requests, passwords, or tokens.

CORS

Cross-Origin Resource Sharing. A browser security rule that blocks a webpage on one domain from freely calling an API on another domain unless that API explicitly allows it. It's a common source of confusing errors the first time you connect a frontend to a backend running on a different port or domain.

least privilege

Give each piece of the system only the access it actually needs. A function that only ever reads users shouldn't be using database credentials that can also delete tables.

Beyond the basics

For when the fundamentals feel comfortable and the next layer of "why do real systems look like this" becomes interesting:

caching

Storing the result of an expensive operation so the next request can skip redoing the work. If GET /users is called a thousand times a minute and the data rarely changes, caching the response for a few seconds can dramatically cut database load.

load balancing

When one server can't handle all the traffic, you run several copies and put a load balancer in front to spread requests across them. This only works cleanly because requests are stateless — any server can answer any request.

horizontal vs. vertical scaling

Vertical scaling means making one server bigger (more CPU/RAM). Horizontal scaling means running more servers side by side. Most large systems lean horizontal, since there's a ceiling on how big one machine can get.

monolith vs. microservices

A monolith is one codebase handling everything — simpler to build and reason about early on. Microservices split the system into smaller, independently deployed services (a "users service," an "orders service," etc.) — more flexible at large scale, but with real added complexity in coordinating them.

message queues

For work that doesn't need to finish before responding (sending a confirmation email after signup), the request can drop a message in a queue and respond immediately, while a separate worker process handles the email afterward.

rate limiting

Capping how many requests a single user or client can make in a given window, to protect the backend from being overwhelmed — by mistake or on purpose.

Glossary

HTTP

The protocol web browsers and servers use to send requests and responses to each other.

JSON

JavaScript Object Notation — a plain-text format for structured data, e.g. {"id": 1, "name": "Asha"}.

endpoint

One specific URL + method combination an API exposes, e.g. GET /users.

ORM

Object-Relational Mapper — translates code calls into SQL behind the scenes.

request / response

A request is what the client sends; a response is what the server sends back.

schema

The defined structure of a database — which tables exist, which columns they have, and what type of data each column holds.

payload

The actual data inside a request or response — as opposed to the headers and metadata wrapped around it.

CRUD

Create, Read, Update, Delete — the four basic operations almost every resource in an API supports, in some form.

latency

The delay between sending a request and getting a response. Distance, server load, and database speed all add to it.

token

A piece of data (often a long random-looking string) issued after login, sent with later requests to prove who you are without re-sending your password every time.

// build the real thing

Build the GET /users endpoint, for real

Everything in the animated tutorial — the route, the function, the SQL, the JSON — was a simulation. This is the same flow, written as actual Node.js + Express code you can run on a real machine. Work through it top to bottom; each step builds on the last.

📦 Before you start: install Node.js from nodejs.org (it comes with npm). Then open a terminal in an empty folder for this project.
01Set up the project

Every Node project starts with a package.json — a file that tracks the project's name and its dependencies. Then we install Express, the library that does the routing and request/response handling for us.

terminal
# inside your empty project folder npm init -y npm install express

That's it — node_modules and a package.json should now exist in the folder.

02Create the server

This is the "server" box from the diagram — a program that starts up and listens for requests. Create a file called server.js:

server.js
const express = require('express'); const app = express(); app.listen(3000, () => { console.log('Server running on http://localhost:3000'); });

Run it with node server.js. You won't see much yet — there are no routes — but the server is alive and listening.

03Add the route

This is the "Route" box. We tell Express: when a GET request comes in for /users, run this function.

server.js — add above app.listen()
const mockUsers = [ { id: 1, name: 'Asha Patel', role: 'admin' }, { id: 2, name: 'Leo Martins', role: 'editor' } ]; app.get('/users', (req, res) => { res.json(mockUsers); });

Restart the server, then open http://localhost:3000/users in a browser. You should see the JSON array straight back — that's the entire request lifecycle from the diagram, just without a real database yet.

04Pull the route's logic into its own function

Right now the route and the function are squashed together. In a real project they're separated — this is the "Function" box. Create controllers/userController.js:

controllers/userController.js
const mockUsers = [ { id: 1, name: 'Asha Patel', role: 'admin' }, { id: 2, name: 'Leo Martins', role: 'editor' } ]; exports.getUsers = (req, res) => { res.json(mockUsers); };
server.js — replace the inline route
const userController = require('./controllers/userController'); app.get('/users', userController.getUsers);

Same result in the browser, but now the route file only does routing, and the controller only does logic — exactly the Route → Function split you watched in the animation.

05Add a middleware (a tiny logger)

Middleware runs before a request reaches its function. Here's the simplest possible one — it just logs every request, then passes control along with next().

server.js — above your routes
app.use((req, res, next) => { console.log(`${req.method} ${req.url}`); next(); // without this, the request would hang forever });

Refresh /users and check your terminal — you'll see GET /users logged every time, before the controller ever runs.

06Swap the mock array for a real database (optional next step)

This is the part the animated tutorial simulated with SQL. When you're ready, an ORM like Sequelize or Prisma replaces the mock array with a real query — the controller code barely changes:

controllers/userController.js — with a real ORM
const { User } = require('../models'); exports.getUsers = async (req, res) => { const users = await User.findAll(); res.json(users); };

Notice it's the exact same line you saw highlighted in the diagram's database step — that's not a coincidence, it's the real call.

🧪 try it yourself

Add a second route, GET /users/:id, that returns just one user matching that id from mockUsers — and a 404 if no user matches. (:id in the path becomes req.params.id.)

show one possible answer
